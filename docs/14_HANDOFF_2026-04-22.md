## Answers to the 5 open questions in `13_WM_PRESENTER_SPEC.md`

### Q1 — Full category list
**Categories are dynamic, not hardcoded.** Admin can add/edit/remove categories from a settings UI. Store as a `categories` table or JSONB list on chapter. The spec's example list (Business Service, Manufacturers, Finance, Marketing, Events, Health & Beauty) is illustrative only.

**How to apply:**
- `member_presentation_card.business_category` is a free-text VARCHAR (no enum constraint).
- Build a `chapter_categories` table later if grouping/filtering needs it; for now, distinct values from existing cards drive the dropdown.

### Q2 — Rotation slot size
**Build rotation as an opt-in feature.** Default mode: all 51 members shown every week (auto-hide absent). The reason rotation wasn't run before was admin laziness — automating it removes that excuse, so the system should support it.

**How to apply:**
- `meeting_templates.slide_order` supports two modes per category block:
  - `mode: "all"` → materialize one slide per active member in category (default).
  - `mode: "rotation"` → materialize N members per week, cycling through the category roster across weeks. Store `rotation_state` per category (last-shown member IDs + week index) on `chapter_categories` or a new `category_rotation_state` table.
- Admin toggles mode per category in chapter settings.
- Rotation algorithm: round-robin by `last_presented_at` ascending, slot size configurable per category (default 4).
- `auto_hide_absent_slides()` still runs after rotation selection — absent rotated-in members get hidden, NOT replaced (slot stays empty for the week).

### Q3 — Host-rotation slots
**Source: BNI 7 Core Values** — see `docs/context/BNI_CORE_VALUES.md`. One value featured per week, cycling through values 1–7. Plus Education / Lifelong Learning slot (separate from core value).

**How to apply:**
- Slide template `core_value_weekly` is a `data_bound` slide that picks the value for `(week_number_of_year mod 7) + 1`.
- Content rendered from `BNI_CORE_VALUES.md` (or seeded into a `core_values` table for editability).
- Education slot stays `member_editable` — assigned to the LT Education chair's card.

### Q4 — Special Announcements
**Hybrid: posters (image upload) + structured FP/training rotation.**

Typical content:
- Poster (PNG/JPG) — uploaded by admin, displayed as `file_upload` slide.
- Feature Presentation speaker — auto-pulled from upcoming events table.
- Training schedule — auto-pulled from training/events table.

**How to apply:**
- `slide_type = file_upload` for posters (admin uploads, no member edit).
- `slide_type = data_bound` for FP rotation and training (queries events/training tables).
- Add an `announcements` table later only if free-form text is needed too. Skip in Stage 3.

### Q5 — PPTX conversion
**Async with placeholder.** Upload returns immediately; slide shows "Converting…" until libreoffice finishes; polling client picks up the rendered PDF on next 5s tick.

**How to apply:**
- POST `/api/member/slide/{slide_id}` with PPTX returns `202 Accepted`.
- Background job runs `libreoffice --headless --convert-to pdf`.
- `meeting_slides.uploaded_file_id` is null until ready; presenter renders placeholder when null.

## BNI HUB — Project re-explanation

### What it is
A self-hosted operations platform for BNI Chapter Mahardika (Bandung). Replaces a sprawl of spreadsheets, PPTX files, WhatsApp messages, and BNI Connect manual work with a single web app + database. Hosted at `bnimahardika.qd.je`.

### Stack
FastAPI + PostgreSQL 16 + SQLAlchemy + Alembic + Caddy + Docker. Python 3.11. JWT cookie auth, bcrypt. DeepSeek via OpenRouter for AI copilot.

### Data philosophy
**Fluid hybrid relational + JSONB.** Core identities (members, chapters, meetings) are normalized columns. Variable, evolving fields (visit_data, slide_overrides, contact info) live in JSONB. Lets the schema breathe without an Alembic migration for every new field.

### Modules already built
- **Auth + chapter scoping** — JWT cookies, member self-service login.
- **Roster** — BNI Connect XML SpreadsheetML import, deduplication by name.
- **PALMS import** — weekly stats (attendance, RGI, RGR, visitors, 1-2-1s, CEU, TYFCB) → `palms_snapshot` rows.
- **Visitor import** — visitor report XML, links to inviter member.
- **Members directory** — `/members` route (added during Stage 1+2 as scope drift, kept).
- **Static WM** — old `/wm/{date}` route, will be replaced by Stage 3.

### What we're building next: WM Presenter (Stage 3)
The Weekly Meeting deck — currently a 306 MB / 141-slide PPTX file edited by hand each week — becomes a live web app at `/presenter/{meeting_id}`.

Replaces the PPTX with:
- 5-act structure (Opening, Foundations, Leadership, Weekly Presentations, Closing).
- 5 slide types (static, data_bound, member_editable, external_embed, file_upload).
- Members edit their own card at `/member/card`. Changes appear in next meeting automatically.
- Auto-hide absent members (precedence: admin > manual > auto > visible).
- 5-second polling state machine. Admin clicks "advance" — every screen in the room updates.
- Canva and Google Slides public links rendered in sandboxed iframes.
- PPTX uploads converted to PDF asynchronously, served via pdf.js.

Spec lives in `docs/13_WM_PRESENTER_SPEC.md`. 5 days estimated.
